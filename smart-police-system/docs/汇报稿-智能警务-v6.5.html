第 03 页 — 项目概述 (50s)

1.1 项目背景：传统警务管理存在信息孤岛、流程脱节问题。本项目面向公安基层派出所，建设统一的警务数据中心，覆盖案件、人员、车辆、巡逻、报警、统计六大场景。
1.2 项目目标有五个：第一，建立统一数据中心消除信息孤岛；第二，案件全生命周期电子化，从立案、侦查到结案一站到底；第三，实时警情监控与调度，接警→派发→处置全流程可视化；第四，接入大模型做智能研判和推荐；第五，数据安全与操作可追溯，审计 + 脱敏 + RBAC 三重保障。
系统规模：9 大业务模块、6 类用户角色、7 项 AI 能力。

第 04 页 — 用户角色与权限 (40s)

1.3 用户角色：我们设计了六类角色——

  • 超级管理员 1 人，掌控系统全部功能；账号 admin / 123456
  • 局长/所长 1 人，全局数据查阅 + 报表导出；账号 陈刚 / 123456
  • 民警-接警 1 人，专门负责报警接收 + 处置 + 升级；账号 张明 / 123456
  • 民警-审批 1 人，专门负责案件状态机推进；账号 赵勇 / 123456
  • 民警 4 人，日常办案主力
  • 辅警 1 人，负责报警录入和巡逻打卡；账号 孙强 / 123456

权限采用 RBAC 模型——6 角色 × 60+ 权限点，覆盖菜单/接口/数据三级粒度，做到接口级细粒度控制。

第 05 页 — 技术选型 (50s)

  1. 技术选型。后端是 Java 17 + Spring Boot 3.2.3 核心框架；Spring Security 6 做认证授权和 RBAC；MyBatis-Plus 3.5 做 ORM 和动态 SQL；MySQL 8.0 + Redis 7 持久化和缓存；OkHttp + Apache POI 用于 AI SSE 流式调用和报表导出。
    前端是 Vue 3 + Vite 5 + Element Plus 主流技术栈——Vue 3 组合式 API，Vite 5 构建，Element Plus UI 组件库，ECharts 5 做数据可视化，Axios 做 HTTP，Pinia 做状态管理，Vue Router 做路由和守卫。
    工程化方面：用 Maven 多环境、Git 版本控制、RESTful API 设计、前后端分离部署。

第 06 页 — 系统架构 (60s)

3.1 系统架构采用经典 L5 分层，从下到上：

  • L1 数据存储层 — MySQL 8 + Redis 7
  • L1.5 DAO/Mapper 层 — MyBatis-Plus 动态 SQL 和分页
  • L2 AI 集成层（蓝色突出）— 封装大模型 API 和 Prompt 上下文采集
  • L3 Service 层 — 业务逻辑、事务、权限校验
  • L4 Controller 层 — @RestController + 参数校验 + 统一响应
  • L5 前端展示层 — Vue 3 + Element Plus + ECharts

右侧是五大核心设计：认证流程、权限控制、统一响应、AOP 日志、缓存策略。

第 07 页 — 核心机制流程图 (70s)

3.2 核心机制。认证/鉴权五步走：①登录（用户名/密码）→ ②认证（Spring Security + BCrypt 加密校验）→ ③签发（生成 JWT Token）→ ④请求（前端在 Header 里携带 Token）→ ⑤鉴权（过滤器解析 + RBAC 权限判断）。
RBAC 权限链有三级粒度：菜单级（页面能不能进）、接口级（API 能不能调）、数据级（这条数据能不能看）。数据模型是经典的 User ←→ UserRole ←→ Role ←→ RolePerm ←→ Permission 五张表。
三项配套机制：①Result<T> 统一响应 + 全局异常拦截，错误码一致；②@OperationLog AOP 切面自动记录操作人、IP、耗时；③Redis 管理 Token 黑名单 + 失败计数 + 接口限流。

第 08 页 — 数据库设计 (60s)

  1. 数据库设计。规模：30 张业务表、21 张含演示数据、340+ 条演示记录，覆盖 9 个功能模块。
    核心表组关系：
  • sys_user ↔ sys_role 多对多（用户角色）
  • alarm_record 一对多 alarm_dispatch（警情派发）
  • case_info 一对多 case_progress / case_evidence / case_suspect（案件四条线）
  • patrol_schedule 一对多 patrol_task → patrol_checkin（排班 → 任务 → 打卡）
  • equipment_info 一对多 equipment_borrow（装备借用）

右侧数据安全六项设计：①密码 BCrypt 加盐哈希、不可逆；②身份证/手机号仅存后 4 位做脱敏；③列表脱敏显示避免敏感信息批量泄露；④统一基础字段（id / created_at / updated_at / is_deleted）；⑤逻辑删除保留数据可追溯性；⑥小写蛇形命名统一编码规范。
底部 4 个安全标签：BCrypt / 脱敏 / 逻辑删除 / 蛇形命名。

第 09 页 — 九大功能模块 (40s)

  1. 九大功能模块沿访问路径排列——这其实也是用户实际操作的导航顺序：
    ①警情驾驶舱（首页、看板）→ ②报警受理（接警、派发、处置）→ ③案件管理（立案、证据、流程）→ ④人员档案（重点人员、违法记录）→ ⑤车辆管理（登记、违章、布控）→ ⑥巡逻调度（排班、任务、打卡）→ ⑦警力资源（警员、装备、考勤）→ ⑧统计分析（趋势、对比环比）→ ⑨系统管理（用户、角色、权限）。
    体量：15 个 Vue 页面、95 个 RESTful API、38 个前端文件、121 个 Java 文件。

第 10 页 — 警情驾驶舱（首页）(40s)

5.1 警情驾驶舱是系统的"首页一屏总览"，给值班员和所长看全局。

右半屏是实际运行截图：近 7 天接警趋势折线图（早上 8 点和晚上 9 点两个高峰）、案件类型分布饼图、最新警情列表（按时间倒序），加载有骨架屏、失败有重试按钮。

第 11 页 — 报警受理 + 案件管理 (60s)

5.2 报警受理 + 案件管理。
左侧报警页面：演示数据有 11 条接警记录，支持按警情类型和紧急程度（普通/较大/重大/特别重大）筛选，列表显示报警人、地址、接警人、处警结果。状态机覆盖全程：待派发 → 已派发 → 已出警 → 已处置 → 已归档，每一步都有操作记录。

v6.5 重点新增的 4 项接警能力：
① 派发 4 步事务——校验警情 / 校验警员在岗 / 写派发记录 / 更新警情，事务提交后异步触发 AI 装备推荐，不阻塞主流程
② "我的警情" radio——报警页顶部切换"全部 / 我的警情"，后端只返回分配给自己的警情
③ "现场打卡"按钮——PUT /api/alarm/{id}/arrive，行操作列 v-if status===2 时显红色图标
④ "升级案件"按钮——caseApi.create(caseData) + alarmApi.close()，自动生成 AJ 编号

右侧案件页面：演示 7 件案件，4 字段筛选（案件编号、状态、报案人、报案时间），每行显示案件类型（刑事/治安/行政）、立案民警、办案进度。重点：案件详情抽屉（右侧滑入 650px 宽）含 3 子页签——侦查进展 / 证据材料 / 嫌疑人。
状态机：立案 → 侦查中 → 已移送 → 已结案（v-if status==='investigating' 时底部显示"审批通过"和"移送检察院"两按钮）。

5.3 人员档案 + 车辆管理。
左侧人员档案：10 个重点人员记录，列表字段做了脱敏——身份证号显示前 6 + 后 4 共 13 位、中间用星号代替；手机号前 3 + 后 4 共 11 位、中间 4 位打码。5 种标签分类（涉毒/在逃/重点上访/前科/精神障碍），点人员进去看违法记录时间线。

第 15 页 — AI 技术要点 (70s)

6.1 AI 技术要点——讲讲我们怎么把大模型工程化落地的，分 4 块：
①动态 Key 管理——API Key 走 Redis 动态管理，前端配置页实时输入，不写死在代码里。这页截图就是管理员在 /system/ai-config 配置 Key 和模型名（mimo-v2.5-pro），保存即生效。
②SSE 流式输出——后端用 Spring SseEmitter 异步、OkHttp 逐行读取大模型返回；前端 fetch + ReadableStream 边收边渲染，体验和 ChatGPT 一样一个字一个字地打字。
③Prompt 工程——上下文压缩到 800 token（避免成本爆炸），结构化数据拼接，system / user 双角色注入。
④熔断降级——大模型不可用时灰度入口，主业务流程不受影响，统一兜底提示"重试"。
底部数据：6 个 AI API、800 token 上下文上限、SSE 流式、多模态支持。

第 16 页 — 关键技术亮点 (60s)

  1. 关键技术亮点，分 5 大块：
    ①Security + JWT——BCrypt 密码加密、HS256 签名、5 次失败锁定 30 分钟、Token 黑名单支持主动退出
    ②RBAC 权限控制——6 角色 × 60+ 权限点、接口级 @PreAuthorize、菜单动态加载、无权限直接 403 跳转
    ③SSE 流式响应——SseEmitter 异步、OkHttp 解析 data: 块、fetch + ReadableStream 前端渲染
    ④数据安全——敏感字段截断存储、SQL 预编译防注入、文件路径防穿越 + 类型校验
    ⑤工程化——统一响应 + 全局异常、AOP 操作日志、逻辑删除 + 自动填充
    底部是技术选型原则：成熟稳定 + 社区活跃 + 团队熟悉 + 业务匹配。

备用补充（如果老师追问）

追问 1：为什么选 Spring Boot 3 而不是 2？

2.7 已经 EOL，3.2 是当前活跃版；3.x 原生支持 GraalVM Native Image、虚拟线程（JDK 21）和 Observability，对后续性能优化有空间。

追问 2：SSE 不用 WebSocket 吗？

警务业务是服务端单向推送为主（AI 输出、报警通知），SSE 走 HTTP/HTTPS 简单、穿代理、免握手；WebSocket 双向但运维成本高。当前场景 SSE 更合适，未来要双向再加 WS。

追问 3：RBAC 怎么做到"接口级"？

在 Controller 方法上加 @PreAuthorize("hasAuthority('case:create')") 注解；权限字符串存在 sys_permission 表，启动时加载到 Spring Security 的 GrantedAuthority。每个用户登录时查自己 60+ 权限点，写进 JWT 即可。

追问 4：AI 用了哪个模型？成本怎么算？

当前配置是 mimo-v2.5-pro（截图里能看到），系统支持动态切换模型。上下文压到 800 token、缓存常见问法，单次研判约 0.01-0.05 元。

追问 5：30 张表怎么设计的？

6 大业务表组：用户/角色/权限（5 张）、警情/案件/证据/嫌疑人（8 张）、人员/车辆/违法（6 张）、巡逻/任务/打卡/排班（5 张）、警员/装备/考勤（4 张）、统计/审计/字典（2 张）。所有表统一基础字段 + 逻辑删除 + 蛇形命名。

追问 6：演示数据怎么来的？

21 张表有演示数据（340+ 条）：其中人员档案 10 条、车辆 6 条、案件 7 件、警情 11 条、警员 8 人、巡逻任务 6 个、排班 15 条——都是贴近真实基层派出所业务写死的。开学初我们组去派出所实习观察了一周才编的。
