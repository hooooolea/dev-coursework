## 1. 项目概述

### 1.1 项目背景

传统警务工作依赖纸质台账、分散 Excel 和人工派单，存在以下痛点：

- **信息孤岛**：案件、人员、车辆、警员、装备等数据分属多个系统，跨模块关联查询困难
- **流程不可追溯**：报警、派发、立案、侦查、结案全凭人工登记，缺乏时序记录
- **决策缺乏支撑**：局长/所长层缺少实时统计、趋势对比、风险预警
- **重复录入**：警员档案、人员档案、案件嫌疑人等信息在多模块间重复维护

本项目面向中小型公安/派出所的日常警务场景，构建一套**统一数据中心 + 流程闭环 + 智能辅助**的数字化平台。

### 1.2 项目目标

| 维度 | 目标 |
|------|------|
| 数据 | 建立 30 张表的统一警务数据中心，消除信息孤岛 |
| 流程 | 案件全生命周期电子化（报警 → 立案 → 侦查 → 移送 → 结案） |
| 监控 | 实时警情驾驶舱 + 7 天趋势 + 类型分布 |
| 智能 | 接入大模型，实现排班推荐、装备推荐、警情研判、语音接警 |
| 安全 | JWT 无状态认证、RBAC 细粒度授权、操作可追溯 |
| 体验 | 加载骨架屏、空状态、二次确认、脱敏显示、移动端可用 |

## 2. 需求分析

### 2.1 功能需求

按照业务域划分，本系统包含 **9 大业务模块 + AI 智能辅助层**：

```
┌────────────────────────────────────────────────────┐
│                  AI 智能辅助层                       │
│   排班推荐  装备推荐  警情研判  AI助手  语音/视频    │
├────────────────────────────────────────────────────┤
│  警情驾驶舱  报警受理  案件管理  人员档案  车辆管理   │  业务模块
│  巡逻调度  警力资源  装备管理  统计分析              │
├────────────────────────────────────────────────────┤
│               系统管理层（用户 角色 权限 部门）       │
└────────────────────────────────────────────────────┘
```

### 2.2 非功能需求

- **性能**：单表数据 10 万级，列表分页 + 索引；接口平均 RT < 300ms
- **安全**：JWT 无状态、BCrypt 密码、RBAC 菜单/接口/数据三级粒度、SQL 预编译
- **可用**：7×24 连续运行、关键路径熔断降级、Redis 缓存热点数据
- **可维护**：统一响应、OperationLog AOP、统一异常、代码规范
- **可扩展**：模块解耦、按业务域分 Maven 包、按页路由懒加载

### 3.5 数据库设计原则

- 统一基础字段：`id / created_at / updated_at / is_deleted / version`
- 逻辑删除（`is_deleted=0` 默认值），保留可追溯性
- 敏感字段脱敏存储（身份证/手机号仅后 4 位）
- 命名规范：小写蛇形（snake_case）
- 表前缀按模块：`sys_ / alarm_ / case_ / person_ / vehicle_ / patrol_ / officer_ / equipment_`
- 索引策略：高频查询字段（user_id、status、time）单独索引

## 4. 详细设计

### 4.1 认证流程

```
①登录（用户名/密码）
   ↓
②认证（Spring Security + BCrypt 加密校验）
   ↓
③签发（生成 JWT AccessToken + RefreshToken）
   ↓
④请求（前端在 Header 里携带 Token）
   ↓
⑤鉴权（JwtAuthenticationFilter 解析 + RBAC 权限判断）
```

JWT 包含 userId + username + type（access/refresh），签名密钥 HS256，过期时间 2 小时，RefreshToken 7 天。Redis 黑名单支持主动登出。

### 4.2 RBAC 权限模型

**三级粒度**：
- 菜单级：Vue Router 守卫 + 后端菜单接口
- 接口级：`@PreAuthorize("hasAuthority('case:view')")` 注解
- 数据级：Service 层根据 userId/deptId 过滤

**5 张核心表**：
```
User ←→ UserRole ←→ Role ←→ RolePerm ←→ Permission
```

### 4.3 统一响应与全局异常

```java
Result<T> { code, message, data, timestamp, ok }
```

`@RestControllerAdvice` 全局拦截 `BusinessException`（业务异常）和 `Exception`（系统异常），统一包装为 Result。错误码规范：200 成功 / 400 参数 / 401 未登录 / 403 无权限 / 404 资源 / 500 系统。

### 4.4 AOP 操作日志

`@OperationLog(module="案件管理", action="立案")` 注解 + 切面自动记录：操作人 / IP / 耗时 / 入参 / 返参。落 `sys_operation_log` 表。

### 4.5 SSE 流式响应

`SseEmitter` 异步推送 + OkHttp 逐行读取大模型返回的 `data: {...}\n\n` 块，前端 `fetch + ReadableStream` 边收边渲染。

### 4.6 AI 模块设计

```
AI 配置（AIKey Redis 动态管理）
   ↓
AI 服务（Prompt 工程 + 上下文压缩到 800 token）
   ↓
AI 调用（OkHttp HTTP / SSE 流式）
   ↓
AI 降级（熔断 + 兜底提示）
```

**7 项 AI 能力**：
1. 智能排班推荐（SSE）
2. 装备推荐（JSON）
3. 警情研判（SSE）
4. AI 助手（SSE）
5. 语音接警（ASR）
6. 视频分析（Omni 多模态）
7. TTS 语音播报

> **v1.1 实际状态**：装备推荐需配置 API Key 才能使用（未配置时 30s 超时）；警情研判 / AI 助手 / 排班推荐 / 视频分析页面有入口但**后端需 API Key**；语音接警需浏览器麦克风权限。

## 5. 功能实现

### 5.2 报警受理（v1.1 重大更新）

**状态机**：
```
接警录入 → 派发警员 → 警员接收 → 到达现场 → 处置中 → 关闭
```

**v1.1 新增能力**（2026-06-15 落地）：

1. **派发 4 步事务 + @Async AI 装备推荐**（`AlarmServiceImpl.java:93 dispatch`）
   ```java
   @Transactional(rollbackFor = Exception.class)  // 类级别
   public void dispatch(Long alarmId, Long officerId) {
       // 1. 校验警情
       // 2. 校验警员在岗
       // 3. 写派发记录
       // 4. 更新警情 status 1→2
       // 5. 异步触发 AI 装备推荐（@Async，事务提交后执行）
       triggerEquipRecommend(alarm, officer);
   }
   ```

2. **"我的警情" radio**（`AlarmView.vue` 顶部）
   - el-radio-button 双标签（全部 / 我的警情）
   - 后端 `GET /api/alarm/my-tasks` 返当前用户分配给自己的警情
   - 实际测试：zhangming 登录后"我的警情"显示 2 条（BJ20260615001 + BJ20260610001）

3. **"现场打卡"按钮**（`AlarmView.vue:53`）
   - 行操作列 v-if=`row.status === 2` 显红色 link button
   - 调用 `arriveByAlarm` API → 后端 `PUT /api/alarm/{id}/arrive` 更新 dispatch.status 1→3
   - 实际测试：zhangming 打卡成功，操作提示"操作成功"

4. **"升级案件"按钮**（`AlarmView.vue:54`）
   - 行操作列 v-if=`row.status === 1` 显绿色 link button
   - 调 `caseApi.create(caseData)` + `alarmApi.close(row.id, '已升级为案件')`（**前端串两步非事务**）
   - 实际测试：zhangming 升级 BJ20260612001 → AJ20260615001 成功

> **v1.1 已知 BUG**：`upgradeToCase` 不传 `relatedAlarmId`（`CaseInfo.relatedAlarmId` 字段虽存在但前端未填），BJ→AJ 关联未真正落地。

### 5.3 案件管理（v1.1 重大更新）

**状态机**：
```
立案（pending，未实现） → 侦查中（investigating） → 已移送（transferred） → 已结案（closed） / 撤案（cancelled）
```

> **v1.1 实测状态**：实际只有 `investigating → transferred → closed / cancelled` 三段，**无 pending 状态**（createCase 直接写 investigating）。

**v1.1 新增能力**：

1. **案件详情抽屉**（`CaseView.vue:117 el-drawer`）含 3 Tab：
   - 侦查进展（`CaseProgress` 表）
   - 证据材料（`CaseEvidence` 表）
   - 嫌疑人（`CaseSuspect` 表）

2. **状态机推进按钮**（`CaseView.vue:51`）
   - 行操作列 v-if=`row.status === 'investigating'` 显绿色"推进" link button
   - 调 `caseApi.updateStatus(row.id, 'transferred', '审批通过，移送检察院')`
   - 实际测试：zhaoyong 推进 AJ20260610002（investigating→transferred）成功

> **v1.1 已知 BUG**：
> - 后端 `CaseSuspectController` 和 `CaseController` 都注册了 `/api/case/{id}/suspect`，Spring MVC 报 `Ambiguous handler methods mapped` 500，**案件详情抽屉打开时 suspect API 失败**
> - 案件管理页**无独立"待审批" Tab**（仅 el-select 筛选项的 pending）
> - 案件管理页**无"审批通过"独立按钮**（"推进"按钮直接 investigating→transferred 一段）
> - 案件管理页**无"检察院接收"独立状态**
> - 案件管理页**无"立案"独立状态**（createCase 直接写 investigating）

### 5.4 人员档案 + 车辆管理

| 模块 | 关键设计 |
|------|----------|
| 人员 | 身份证+手机号脱敏存储后 4 位；分类标签（普通/重点关注/在逃/涉毒/社区矫正） |
| 车辆 | 车牌 + VIN + 品牌型号；状态机（正常/可疑/布控中/在逃/已扣押）；布控/解控可追溯 |

### 5.5 巡逻调度 + 排班

- **任务管理**：CRUD + 状态（待派发/已接收/进行中/已完成/已取消）
- **签到打卡**：到达现场时上传位置 + 时间 + 简述
- **周视图排班**：早/中/夜 × 周一~周日 网格，单元格内显示当班警员头像
- **出勤统计**：按时段统计完成率

### 5.6 警力资源 + 装备管理

- **警员档案**：警号、警种、警衔、职务、擅长领域、在职状态
- **绩效考核**：出勤 + 案件 + 奖励 − 违规 = 总分，自动评级（优秀/合格/不合格）
- **装备物资**：10 大类（通信/取证/防护/照明/警械/检测/医疗/侦查/交通/宣传），借还留痕

### 5.7 统计分析

- **案件统计**：按月立案趋势 + 类型饼图 + 状态汇总
- **警力效能**：在岗率、状态分布、活跃排行 Top10
- **同比环比**：立案总数、已结案、破案率，计算公式 `((本期 - 上期) / 上期) * 100%`
- **Excel 导出**：Apache POI 生成 `.xlsx`，前端 `<a download>` 直接下载

> **v1.1 实测状态**：驾驶舱"警员工作量排行"卡片**未实现**（虽有警力效能 Top10 描述但 dashboard 视图未展示此卡片）。

### 5.8 AI 智能辅助（v1.1 重大更新）

| # | 功能 | 入口 | 模式 | v1.1 实测状态 |
|---|------|------|------|---------------|
| 1 | 智能排班推荐 | 巡逻任务派发 | SSE | 需 API Key |
| 2 | 装备推荐 | 派发后自动 / 报警页右上角 | JSON | **实测**：未配 Key 时 30s 超时；有 Key 时返 must + suggested |

| 4 | AI 助手 | 全局悬浮窗 🤖 | SSE | 页面存在，需 API Key |
| 5 | 语音接警 | 报警录入 | ASR | 需浏览器麦克风权限 |
| 6 | 视频分析 | 证据上传 | Omni | 需 API Key + base64 编码 |
| 7 | TTS 播报 | 紧急警情 | TTS | 需 API Key |

> **v1.1 重要说明**：
> - 派发后**自动触发** AI 装备推荐（`@Async`），但日志会显示失败（`装备推荐失败 alarmId=X, AI 服务暂不可用`），不影响主流程
> - **v1.1 新增**：`@Async triggerEquipRecommend` 方法（`AlarmServiceImpl.java:123`），在 dispatch 主事务后异步执行
> - "AI 警情研判"独立功能**未实现**——本项目 AI 模块仅实现"装备推荐"作为派发辅助；研判 / 助手 / 排班有前端页面但**未真正集成到业务流**

### 8.1 完成度

- **30 张表 + 95 个 API + 121 个 Java 文件 + 38 个前端文件**（v1.0 数 +5 API）
- **9 大业务模块全部实现**，AI 智能辅助部分集成

- **340+ 条演示数据**，覆盖报警 → 结案全流程

### 8.2 技术亮点

- 4 步派发事务 + @Async AI 装备推荐
- "我的警情" + 现场打卡 + 升级案件完整接警链路
- 案件详情抽屉 3 Tab（进展/证据/嫌疑人）
- 状态机推进 investigating → transferred

### 8.3 不足与改进方向（v1.1 真实清单）

1. **后端 BUG**：`CaseSuspectController` 与 `CaseController` 都注册 `/api/case/{id}/suspect`，需删除其中一个
2. **BJ→AJ 关联未真正落地**：`upgradeToCase` 不传 `relatedAlarmId`
3. **状态机缺 `pending` 状态**：createCase 直接写 investigating，建议加立案审批流
4. **缺"待审批"独立 Tab + "审批通过"独立按钮**：当前仅 el-select 筛选项 + 推进按钮
5. **缺"检察院接收"独立状态**
6. **缺"警员工作量排行"卡片**
7. **AI 智能研判独立功能未实现**（仅装备推荐）
8. **`Officer.currentLoad` 字段未返**（派发弹窗无法按"未满负荷"过滤）
9. **新加 zhangming / zhaoyong 账号未分配角色**（authorities=[]）

### 8.4 个人收获

- **Spring Boot 全栈工程能力**：从 0 到 1 跑通 L5 分层架构
- **Vue 3 组合式 API 实战**：ref / reactive / watch / computed / 组件通信
- **AI 大模型集成**：SSE 流式 + Prompt 工程 + 动态 Key 管理 + 熔断降级
- **工程化能力**：统一响应 / AOP 日志 / 异常处理 / 文档规范
